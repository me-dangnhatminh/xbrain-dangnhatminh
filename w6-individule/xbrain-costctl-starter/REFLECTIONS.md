# Reflections — G5 costctl

## 1. `clean --apply` blast radius

**Q:** If you accidentally ran `clean --tag Environment=dev --apply` in an account shared with another team, what would you have wanted in place to limit damage?

**A:** Several safeguards would help:

- **Require a second confirmation with resource count** — before applying, print exactly how many resources will be terminated and require typing the count or "yes" (not just "y"). The current `--apply` flag alone is too easy to pass accidentally in a script.
- **Scope restriction via account/OU tags** — tag resources with `Team=<team-name>` and have `clean` refuse to touch resources that belong to another team. This is a convention-based guard but very effective.
- **Resource protection** — use EC2 termination protection (`DisableApiTermination=true`) and deletion protection on RDS. `clean` would fail on protected resources rather than silently deleting them.
- **Time-delayed apply with undo window** — instead of immediate termination, mark resources for deletion (e.g., stop instances + tag with `scheduled-delete=<timestamp>`). A separate cron job actually terminates after 24h. This gives a window to recover from mistakes.
- **IAM boundary** — scope the CLI's IAM role so it can only terminate resources tagged with your team's cost center. Even if the code has a bug, AWS itself blocks the blast radius.

The key insight: **defense in depth**. No single safeguard is enough — combine code-level checks (dry-run default, confirmation) with infrastructure-level guards (termination protection, IAM boundaries).

## 2. AI assistance

**Q:** What fraction of code came from AI tools unmodified? Which parts did you actively modify, why?

**A:** Approximately 90% of the implementation code was generated by Claude Code (Claude Opus). The code was generated based on the detailed docstrings/specs already provided in the starter template.

- **Architecture decisions** — choosing to implement all 5 commands (list + cost + terminate + tag + clean) rather than the minimum 3 required.
- **Error handling edge cases** — the S3 `get_bucket_tagging` ClientError handling needed understanding of how moto simulates the missing-tags scenario vs real AWS behavior.

The AI excelled at: boto3 API boilerplate, translating docstring specs into working code, understanding the test assertions to infer expected return shapes.

The human (me) was essential for: verifying the tests actually pass, running against real AWS to validate, and writing these reflections with genuine understanding of what the code does.

## 3. Multi-account

**Q:** To run `costctl` against 100 AWS accounts, what changes?

**A:** The current CLI assumes a single set of credentials. For 100 accounts:

1. **Cross-account IAM roles** — create a `CostCtlReadOnly` role in each account that trusts a central "management" account. The CLI would `sts.assume_role()` per account before making API calls.
2. **Profile loop or config file** — either iterate over AWS CLI profiles (`--profile account-042`) or maintain a YAML/JSON manifest mapping account IDs to role ARNs.
3. **Aggregated output** — add `--account` column to all output, or generate per-account CSV files. For `cost`, aggregate across accounts into a single report grouped by account + service.
4. **Parallelism** — 100 sequential assume-role + API calls would be slow. Use `concurrent.futures.ThreadPoolExecutor` to query accounts in parallel (respecting API rate limits with exponential backoff).
5. **Caching** — Cost Explorer data doesn't change frequently. Cache results locally with a TTL to avoid redundant API calls during iterative analysis.
